Vibe Coding Project
The Vibe Coding Project trains models to build client‑side web applications. This guide explains how to create high‑quality prompts, rubrics, and metadata to teach models real‑world skills like state management, event handling, and layout correctness.
Every training sample consists of:
A prompt describing what to build.
A rubric defining how the output will be graded.
Metadata specifying the task’s category, difficulty, and allowed libraries.
📝 Task Overview
You are responsible for the prompt, rubric, and metadata. After the model attempts the task, the evaluation pipeline generates additional artifacts.

💎 Golden Rule
Tasks are evaluated by vision-language models using only a single screenshot and the submitted HTML/JS code.
Prompts should describe interactive, stateful client-side applications
Applications may include buttons, inputs, controls, and state changes
Tasks should not be purely static pages with only text or images
Rubric items must be checkable from a screenshot and code inspection only, without requiring interaction. Criteria that would require clicking, typing, or exploring UI to validate cannot be evaluated by vision language models and therefore will result in task failure. This describes how tasks are evaluated, not the type of application you should build.
No hidden states: Do not require evaluators to explore unseen states, pages, or flows
🏗️ End‑to‑End Workflow
Creating a training sample involves a structured three-step process:

1. Define the Concept
   Choose a client‑side idea (single page or small React app) that fits the target difficulty. Outline core features and ensure the concept is original. Decide on the category (e.g., game, data visualization, or tool).
2. Write the Prompt
   Describe the application using clear, self‑contained requirements. Vary the style and length of prompts across the dataset to improve model robustness.
3. Write the Rubric
   Craft 8 -20 atomic, tagged, pass/fail rubric items that correspond directly to the prompt. If you exceed this (especially past 20), prioritize: keep only the highest-signal rubrics and consolidate lower-impact ones.
   Start each item with an action verb.
   Ensure every item is verifiable from either a screenshot or the code.
   Model Failure Requirement
   Purpose
   Our goal is to intentionally calibrate task difficulty so that model performance degrades as difficulty increases.
   Target Pass Rates by Difficulty
   Design prompts and rubrics such that model pass rates fall within these ranges:
   Easy: 70–100% pass rate
   Medium: 40–70% pass rate
   Hard: 0–40% pass rate
   A task is considered well-calibrated when its observed pass rate aligns with its intended difficulty band.
